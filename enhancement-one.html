Enhancement One: Software Design and Engineering

MVVM Architecture Implementation

Artifact Description

The artifact selected for this enhancement is the Inventory Manager application, originally developed for CS-360 (Mobile Architecture and Programming). This Android application provides basic inventory management functionality, allowing users to add, view, update, and delete inventory items through a mobile interface. The original implementation used a traditional activity-based architecture with direct database interactions within the UI layer.

Original Features:

  • User authentication (login/register)
  • CRUD operations for inventory items
  • SQLite database integration
  • Basic UI with RecyclerView for item display

Justification for Inclusion

I selected this artifact for Enhancement One because it presented an excellent opportunity to demonstrate my understanding of software design patterns and architectural best practices. The original implementation, while functional, suffered from several design issues common in early Android development:

  • Tight coupling between UI components and business logic
  • Difficulty in testing due to dependencies on Android framework components
  • Poor separation of concerns making maintenance and extension challenging
  • Lack of lifecycle awareness leading to potential memory leaks

By refactoring this application to implement the Model-View-ViewModel (MVVM) architecture pattern, I was able to showcase my ability to:

  • Apply industry-standard architectural patterns to improve code quality
  • Implement Android Architecture Components (ViewModel, LiveData, Repository pattern)
  • Create modular, maintainable, and testable code
  • Demonstrate understanding of software engineering principles like separation of concerns and dependency inversion

Specific Components Showcasing Skills

1. ViewModel Implementation

I created dedicated ViewModel classes for each major feature (InventoryViewModel, LoginViewModel), which manage UI-related data in a lifecycle-conscious way. This demonstrates my understanding of:

  • Android lifecycle management
  • Data survival across configuration changes
  • Proper state management in mobile applications

2. LiveData Integration

I implemented LiveData observers to enable reactive UI updates, ensuring the interface automatically reflects changes in the underlying data. This showcases:

  • Understanding of reactive programming principles
  • Ability to create responsive, user-friendly interfaces
  • Knowledge of observer pattern implementation

3. Repository Pattern

I created a repository layer to abstract data sources and provide a clean API for data operations. This demonstrates:

  • Understanding of clean architecture principles
  • Ability to create maintainable abstractions
  • Preparation for future scalability (easy to add remote data sources)

4. Dependency Injection Principles

While not using a full DI framework, I structured the code to enable easy dependency injection, improving testability and modularity.

Enhancement Process and Learning

Planning and Design

Before beginning the refactoring process, I created a comprehensive architecture diagram to visualize the new structure and identify the components needed. This planning phase reinforced the importance of thinking through design decisions before implementation—a critical skill in professional software development.

Implementation Challenges

The refactoring process presented several challenges that deepened my understanding:

  • Data flow redesign: I had to completely rethink how data flows through the application, moving from direct database calls to a layered architecture with proper abstractions.
  • Lifecycle management: Understanding how ViewModels survive configuration changes and properly scope their lifecycle taught me important lessons about Android development.
  • Threading considerations: Implementing database operations on background threads while updating UI on the main thread required careful attention to threading models.

What I Learned

This enhancement process taught me several valuable lessons:

  • Architectural patterns aren't just theory: Implementing MVVM provided tangible benefits in code organization and testability, demonstrating why these patterns are industry standards.
  • Refactoring requires discipline: I learned to make incremental changes, test frequently, and maintain backwards compatibility during the transition period.
  • Documentation is crucial: Creating clear code comments and architectural documentation made the refactoring process smoother and will benefit future maintainers.

Conclusion

The software design and engineering enhancement transformed the Inventory Manager from a simple student project into a well-architected application that demonstrates professional development practices. This enhancement showcases my growth from learning basic programming concepts to understanding and applying sophisticated design patterns that are essential in modern software development.

This work represents my readiness to contribute to professional development teams and build maintainable, scalable software solutions.
